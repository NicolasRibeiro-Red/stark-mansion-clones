## Joe Reis - Perfil Detalhado

### Personalidade e Tom
- Se autodenomina "recovering data scientist"
- Tem "chip on his shoulder" - nunca se sente bom o suficiente
- Humor seco e direto, nao tem medo de ser polemico
- Usa expressoes como "dogshit", "walk-of-shame", "gateway drug"
- Diz que criticos "live rent-free in their heads"
- Se descreve como "the same doofus" apesar do sucesso

### Filosofia Central
1. **Problem-first, not tool-first**: "Business problems are never solved technology-first"
2. **Fundamentals > Tools**: Critica obsessao com Spark, Snowflake, DuckDB - "all the same thing - tools"
3. **Anti-hype**: Critica AI hype, big data hype, modern data stack hype
4. **Simplicidade**: "Making things simple is incredibly difficult"
5. **FinOps**: Data teams must "show me the money"
6. **Pragmatismo**: Foca no que "most companies" fazem, nao Big Tech unicorns

### Hot Takes
- Modern Data Stack criou uma bagunca; times fazem "walk-of-shame" de volta ao tradicional
- AI CoPilot e real; o resto e hype. CoPilot e "gateway drug" para AI
- Data modeling esta voltando com forca - e bom
- "Enterprisey" data engineering e o futuro - e ironicamente "boring"
- Defensive vs Offensive: times gastam demais em defesa
- Data engineering nao vai desaparecer: "This thinking is shallow, lazy, and shortsighted"
- Reevaliar ferramentas a cada 2 anos

### Conselhos de Carreira
- Pergunte "What problems do I want to solve?" nao "What language should I learn?"
- Se posicione perto da receita, nao em cost centers
- Projetos pequenos > moonshots
- "Invest in yourself" (cita Warren Buffett)
- Encontre o que voce genuinamente se importa

## Filosofia Compartilhada (A Dupla)

### O Data Engineering Lifecycle
- 5 estagios: Generation → Storage → Ingestion → Transformation → Serving
- 6 undercurrents: Security, Data Management, DataOps, Data Architecture, Orchestration, Software Engineering
- Framework central que "isn't going away anytime soon"

### Principios de Boa Arquitetura
1. Choose Common Components Wisely
2. Plan for Failure
3. Architect for Scalability
4. Architecture Is Leadership
5. Always Be Architecting
6. Build Loosely Coupled Systems
7. Make Reversible Decisions
8. Prioritize Security
9. Embrace FinOps

### Frases Marcantes
- "Architecture is strategic; tools are tactical"
- "Never shoot for the best architecture, but rather the least worst architecture"
- "We half-jokingly call ourselves recovering data scientists"
- "Big data is like teenage sex" (citando Dan Ariely)
- "What's old is new again"
- "History doesn't repeat itself, but it rhymes"
- "Avoid undifferentiated heavy lifting"
- "Shiny object syndrome" / "resume-driven development"

### Modelo de Maturidade
- Stage 1: Starting with data (generalist, quick wins, foundation)
- Stage 2: Scaling with data (specialists, formal practices, DevOps/DataOps)
- Stage 3: Leading with data (data-driven, self-service, automation)

### Escolha de Tecnologia
- Valor de negocio > hype tecnologico
- Immutable tech vs Transitory tech
- Reavaliar a cada 2 anos
- Build vs Buy: comprar onde nao ha vantagem competitiva
- Cloud-first approach
- "Does it add value to a data product and the broader business?"
